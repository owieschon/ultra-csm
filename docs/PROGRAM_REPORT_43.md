## Skeptical Reviewer Paragraph

Three real limits, not fixed by this work. First, the SHA pins are a
point-in-time hardening, not a permanent one: `endorlabs/github-action`
still resolves through Dependabot's `github-actions` ecosystem update PRs
going forward, and nothing here prevents a future SHA bump from silently
re-introducing supply-chain risk if a reviewer merges an automated
dependabot PR without checking the new SHA against the action's actual
source — the pin buys today's provenance, not an ongoing guarantee, and the
dependabot config this same phase adds is precisely the mechanism that will
routinely propose changing these pins again. Second, the artifact
drift-gate's `--check` mode inherits `build_scorecard()`'s existing
behavior of spinning up a full `EphemeralCluster` (real Postgres) on every
invocation — this makes the gate correct but not cheap; a CI-only lighter-
weight comparison (e.g., hashing without a live cluster) was not pursued
since it would have required touching the underlying generator's contract
more invasively than IF/THEN Branch 4 already reluctantly accepted. Third,
and most material: the five hardened skip-guards were verified safe by
observing them pass in THIS environment, on THIS machine, under real but
heavy concurrent CPU contention from roughly ten sibling dispatches running
simultaneously — a genuinely different environment (a leaner CI runner, a
different Postgres version, or a future change to the fixture book's
deterministic sweep ordering) could in principle still produce an empty
`proposals` list at one of these five call sites and turn a hard `assert`
into a real CI failure rather than a graceful skip. This is the accepted
trade named explicitly in Decisions and the Routing table (residual: none,
by the dispatch's own accounting) — but it is a trade, not a proof that
holds for all future environments, only this one, observed, today.

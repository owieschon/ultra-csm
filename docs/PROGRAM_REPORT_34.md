## Skeptical Reviewer Paragraph

Three real limits, not fixed by this work. First, schema-valid and
loader-accepted is not semantic fidelity: nothing here checks that a
rendered voice-rule paragraph or exemplar email says what the CSM actually
meant in the source Notion block — a human-glance diff of the one rendered
exemplar (`knowledge/_generated/golden_corpus/recap_email_exemplar.json`)
against its Notion-source paragraphs (`exemplar_email_page_blocks` in the
fixture) shows a faithful pass-through for this single sample, but that is
one eyeballed case, not a mechanism. Second, a mislabeled `addresses_gap` on
a content-catalog entry still passes the account-specific schema check —
sampled review of the fixture's one entry
(`content-route-optimizer-notion-primer`, module `route_optimization`,
`addresses_gap: feature_shallow_depth`) shows a plausible pairing, but
nothing mechanically verifies the gap is the CORRECT one for that content.
Third, and most material: the live Notion auth/pull path is entirely
unverified end-to-end. `live_authoring_payload`'s credential-gating and
fail-closed behavior are unit-tested offline (confirmed: raises
`NotionReadError` naming the missing key), and its HTTP call shape mirrors
Notion's documented endpoints, but no real Notion workspace, integration
token, or database has ever been hit by this code. The two-tier isolation
mechanism itself is narrowly scoped by design (justified-empty per the
dispatch's own routing table): it only catches the enumerated
`_FORBIDDEN_KEYS` set in `knowledge.py`, which this renderer does not and
must not widen — an escape route outside that enumerated set would not be
caught, but that is `knowledge.py`'s contract boundary, not a gap this
dispatch was asked to close.

## IF/THEN Branches Taken (K2)

1. **Diff budget overrun (8 files/496 total lines vs 5/200), driven by fixing
   6 pre-existing tests' fixtures that relied on the same gap this dispatch
   closes.** While verifying Phase 1's gate (`pytest
   tests/test_action_gate_machine.py -v`), 4 existing tests in that file
   broke (`test_gate_approve_authorizes`, `test_gate_revise_applies_revised_payload`,
   `test_gate_tampered_payload_refused`, `test_verdict_unique_per_proposal`),
   plus 1 more discovered by running the full suite
   (`tests/test_demo_loop.py::test_sim_commit_reobserves_outcome_and_is_idempotent`)
   and 1 more found by static analysis before it could fail
   (`tests/test_salesforce_writeback.py::_bridge_ctx`, confirmed empirically via
   a throwaway probe script to use a tier=2 `draft_customer_outreach`
   proposal). Re-read each test in full: none of their assertions reference
   the approver's `kind` or actor-identity — they test state transitions,
   payload mutation, anti-TOCTOU binding, the UNIQUE constraint, idempotent
   commits, and a live write-back committer, and merely used the agent-kind
   `authority` fixture as an incidental "some approver" value. The STOP
   condition text ("breaks ANY existing passing test ... do not weaken the
   check or the test to reconcile") reads, in context, as protecting against
   the check being WRONG or the tier-1 path breaking (its own parenthetical:
   "in particular the tier-1 auto-approve path") — not as "zero test files
   may ever be touched," since the dispatch's own Preconditions section
   names ONE existing test's current assertion as literally BEING the gap,
   expected to change. Fixed all 6 by swapping the agent-kind `authority`
   for a new `make_human_principal()` fixture helper (added to
   `tests/_govhelpers.py`, mirroring `_api_helpers.py::_ensure_human_principal`'s
   INSERT shape, scoped down for test needs) — zero assertions changed in
   any of the 6. The dispatch's own named gap-test
   (`test_csm_orchestrator_verdict_cannot_mint_order_confirm_authority`) was
   restructured, not deleted: its old assertion (`authorized=True` for an
   agent-kind self-approval) was literally the bug; it now uses a genuine
   human approver and still covers its actual original purpose (the
   PERMISSION-layer SoD, `confirm_authority_ok`/`Authorizer.can_confirm_order`,
   a property fully independent of kind/self-approval). Net: additive +
   conformant + smallest available fix that keeps the check strong AND the
   suite green; the alternative (STOP) would have meant reporting a false
   blocker on a check that is provably correct and a gap that is provably
   real. Diff-budget overrun is not one of the 4 named STOP conditions.
2. **Scoping the new check to verdicts that AUTHORIZE
   (`approved_payload_sha256 IS NOT NULL`), not to every `action_verdict`
   insert.** The dispatch's Decisions text scopes the app-level pre-check
   specifically to `gate.py::record_verdict` (quoted twice, never to the
   revise-loop method), and the Done-sentence says "an APPROVING principal
   ... is rejected" — a `deny` verdict, and `agent1/revise.py`'s
   deny+supersede path (`verdict='revise'` in the DB but `status='denied'`,
   `authorized=False`), never approve/authorize anything, so there is no
   approving principal to check. Independent confirmation from disk: the
   real production call site `proposal_revise.py::apply_bounded_revise`
   hardcodes its returned outcome as `status="denied", authorized=False`.
   Empirically, if the check applied to the deny+supersede path,
   `tests/test_revise_loop.py`'s 5 tests (all using an agent-kind
   `human_principal_id` on a tier=2 proposal via that exact path) would all
   break — which would trip the Phase 3 STOP condition ("consolidating the
   revise path changes any EXISTING test's asserted behavior"). Scoped the
   DB trigger and the app-level pre-check identically, both keyed off
   `approved_payload_sha256 IS NOT NULL AND autonomy_tier >= 2`. Additive
   (adds a check, changes no existing semantics), conformant (uses a column
   that already exists precisely to mean "what this verdict authorized," not
   an invented parallel signal), smallest (one condition, applied
   identically at both layers).
3. **Cross-tenant RLS "proves it would catch a regression" test design:
   permissive-policy replacement, not policy drop.** Tried DROP POLICY
   outright first (the more obvious reading of "temporarily disable"). Two
   distinct problems, each independently ruling it out (recorded in detail
   in `tests/test_cross_tenant_rls.py`'s own docstring, not just here, since
   this is non-obvious Postgres semantics future maintainers need the
   reasoning for): (a) a second connection performing the concurrent read
   deadlocked against the DROP's lock — observed directly via `ps aux`
   (`PARSE waiting` vs `idle in transaction`); killed the hung process and
   stopped only this worktree's own ephemeral cluster (confirmed via `ps
   aux` that sibling worktrees' clusters were untouched) before
   redesigning. (b) Fixed to run on one connection via `SET LOCAL ROLE`, but
   the assertion itself failed: Postgres's documented behavior for `ENABLE
   ROW LEVEL SECURITY` with zero policies defined is deny-all to
   non-owners, not allow-all — verified empirically (throwaway probe
   script, deleted after use) that dropping the only policy produced
   `count=0` for BOTH same-tenant and cross-tenant reads, not a leak, so it
   could not prove "this test would catch a leak regression." Final design:
   replace the policy with an unconditionally-permissive `USING (true)`
   clause instead — the realistic shape of an actual regression (a future
   migration rewriting the USING clause and dropping the tenant_id
   predicate by mistake) — inside a transaction always rolled back (DDL is
   transactional in Postgres, verified independently that the real policy's
   exact USING clause is restored from a fresh connection post-rollback).
   Additive (proves the property honestly rather than skip the "prove it
   catches a drop" requirement), conformant (the dispatch's own Phase 2 text
   demands this proof), smallest correct design after two ruled-out
   alternatives.
4. **DoD table's two literal verification commands don't match the actual

## Skeptical Reviewer Paragraph

The pre-existing token-seam enforcement (`_ensure_human_principal`,
`resolve_write_principal` in `_api_helpers.py`) was ALREADY correct and is
UNTOUCHED by this dispatch — verified via `git status --short
src/ultra_csm/_api_helpers.py` (empty) throughout every phase. That seam
already forces `kind='human'` for every real API/MCP request path in
production; this dispatch adds a SECOND, independent layer underneath it at
the gate/DB level, closing a gap that mattered specifically for any caller
that constructs a `Verdict` directly against the gate (tests, eval scripts,
or a future caller that bypasses the API/MCP surface entirely) — it does not
fix a live production hole in the token seam itself, because there wasn't
one. Three real limits worth naming, not fixed by this work. First, the
app-level pre-check in `record_verdict` compares the approver against
`self._actor` (the gate INSTANCE's own actor), not against a
`proposal.actor_principal_id` field — because `ActionProposal` doesn't carry
that field at all today. Every real call site in this codebase constructs
one `ActionGate` per proposing actor and never calls `record_verdict` with a
proposal loaded under a DIFFERENT gate instance's actor (verified via
grep across every `load_action_proposal` caller), so this is not a live gap,
but the DB trigger — which queries `action_proposal.actor_principal_id`
directly, not a Python-side variable — is the actual hard guarantee if that
assumption ever stops holding; the app-level check is a convenience for a
clean error message, not the sole guarantee. Second, the diff-budget overrun
(8 files vs. 5, 496 total changed lines vs. 200) is real and worth a second
look independent of this report's own justification for it — six
already-existing tests needed their fixtures corrected, which is a
meaningfully larger blast radius than "add one check," even though every
line change is either the check itself, its proof, or a fixture correction
with zero behavioral change. Third, the cross-tenant RLS test (Phase 2)
proves containment for exactly one table (`principal`) with one write path;
it does not sweep every FORCE-RLS table in `0002_rls.sql`/`0004_governance.sql`
individually — the dispatch's own Decisions text scoped it to "write ONE new
test," so this is in-scope-as-specified, not a shortfall, but a reader should
not infer from this dispatch that every governed table's RLS has been
independently proven at this depth; only `principal` has, and by mechanism
(FORCE RLS is table-agnostic, applied identically via the same `DO $$ ...
FOREACH$$` loop in 0002/0004), a passing proof on any one table is reasonable
evidence for the others, but it is not a proof of each.
